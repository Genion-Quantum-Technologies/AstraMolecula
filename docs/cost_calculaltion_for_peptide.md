这个模型同样使用一个抽象的“**计算单元 (Compute Unit, CU)**”来量化任务的复杂度，使其完全独立于底层硬件。

### **理论计算总量模型 (Peptide 优化)**

该模型的关键在于理解其**迭代和累积**的特性。总计算量是**单次计算的复杂度**与**总计算次数**的乘积。

#### **核心计算公式**

总计算单元 (CU) 的计算公式可以表示为：

```
Total CUs = (总迭代次数 × 每次迭代的评估次数) × (单次评估的复杂度因子)
```

-----

#### **各参数的计算因子详解**

**1. 基础参数 (Base Parameters)**

我们首先定义一个“标准单次计算”的基准。在这里，**“单次计算”指运行一次 Rosetta ddG 评估**。

  * 标准 `peptide_sequence` 长度: **10** 个氨基酸。

我们将一个标准长度为10的多肽进行一次Rosetta评估的计算量定义为 **1 CU**。

**2. 总计算次数 (Total Calculation Runs)**

这部分由两个参数线性相乘得到，代表了核心的 Rosetta 计算总共被调用了多少次。

  * **`n_iterations`**: 优化的总轮数。
  * **`n_rosetta_runs`**: 在**每一轮**迭代中，为了获得稳定的能量评估而重复运行 Rosetta 的次数。
  * **计算**: `总计算次数 = n_iterations × n_rosetta_runs`

**3. 单次评估的复杂度因子 (Per-Calculation Complexity Factor)**

这部分由多肽序列的长度非线性地决定。

  * **`peptide_length`**: `len(peptide_sequence)`，即输入多肽的氨基酸数量。
  * **复杂度因子 (`F_length`)**: 描述了肽链长度如何影响单次 Rosetta ddG 计算的成本。由于原子间相互作用的数量随链长度增长而快速增加，这个关系是**非线性的**。我们使用 **1.5 次方**来模拟这种介于线性和二次方之间的多项式增长关系，这在生物分子模拟中是常见的复杂度近似。
  * **计算**: `F_length = (peptide_length / 10) ** 1.5`
  * **解释**:
      * 一个长度为10的肽链，其复杂度因子为 `(10 / 10) ** 1.5 = 1`。
      * 一个长度为20的肽链，其复杂度因子为 `(20 / 10) ** 1.5 ≈ 2.83`。这意味着单次计算的成本是标准肽链的近3倍，而不是2倍，体现了非线性增长。

-----

#### **最终计算模型汇总**

将以上所有因子整合，我们得到最终的计算模型：

**`Total CUs = (n_iterations × n_rosetta_runs) × (len(peptide_sequence) / 10) ** 1.5`**

其中:

  * `n_iterations`: 用户设定的迭代总次数。
  * `n_rosetta_runs`: 用户设定的每次迭代中 Rosetta 的运行次数。
  * `len(peptide_sequence)`: 初始多肽序列的长度。



这个模型为您提供了一个清晰、可量化的标准来评估不同参数组合下的 Peptide 优化任务的计算成本。
